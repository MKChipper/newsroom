# Newsroom Post Studio Specification

Last updated: 2026-06-13

## Objective

Newsroom should keep its existing one-person newsroom automation, but the main
product experience should feel like a post production studio: the editor can
see what will actually be published, what the agent is making, what the editor
owes, and why a format was chosen.

The core change is moving from "pipeline status first" to "publishable post
first." Ideas, research, drafting, legal review, design, recording, approval,
packaging, and posting remain as the engine underneath.

## Review Findings

1. The current Floor board is pipeline-centric. It shows internal stages and
   counts, but not enough of the post itself: preview, platform, caption,
   asset state, format route, or manual posting package.
2. Format and art direction are not first-class. Stories have loose `platform`
   and `format` strings, while visual direction, hooks, recording needs, and
   generation plans are buried inside script and desk notes.
3. Asset needs are implicit. Recordings are batched, but screenshots, receipt
   crops, product photos, reference images, and "agent can create this" are not
   tracked as explicit requests.
4. The Design Studio is useful, but it starts too late. It edits prompts and
   picks generated images after Gate 1, rather than showing route choices and
   visual obligations during commissioning.
5. Ready-to-post output is mainly delivered to Telegram and `MANIFEST.md`.
   The dashboard needs a Buffer-like Ready queue that exposes caption,
   hashtags, cover text, media, and posting notes before manual posting.

## Product Principles

- Visibility before action: every card should answer "what is this becoming?"
- Separate idea flow from post craft: ideas can remain loose, but drafts and
  approvals should be platform-aware.
- Human obligations must be explicit: if Liz needs to provide a screenshot,
  receipt crop, voice recording, product/reference image, or approval, it
  appears in one place.
- Agent work should be inspectable: generated art, prompts, format rationale,
  and cost should be visible before approval.
- Manual publishing stays: no platform API posting, no automatic live posts.

## Core Objects

### Creative Brief

One brief per story, generated by the story desk and refined by route selection.

Fields:
- `storyId`
- `status`: `drafted`, `selected`, or `superseded`
- `researchSummary`: plain-language summary of the source material and why the
  story exists now.
- `audienceLanguage`: phrases from the source/audience that can shape hooks.
- `editorFocus`: optional note from Liz.
- `createdAt`, `updatedAt`

### Format Route

A route is one concrete way this story could become a post.

Fields:
- `storyId`, `briefId`, `order`
- `title`
- `angle`
- `platform`: `tiktok`, `instagram`, `linkedin`, `meta`, etc.
- `format`: `tiktok_video`, `tiktok_carousel`, `ig_reel`, `ig_carousel`,
  `static`, `meta_ad`, etc.
- `postType`: human-readable format family, e.g. "receipt-led reel",
  "informational carousel", "single-frame comparison".
- `structure`: what the post is made from, e.g. "hook VO + 3 receipt beats +
  payoff".
- `visualTreatment`: art direction, not content copy.
- `assetStrategy`: `agent_can_create`, `needs_liz_assets`, `mixed`, or
  `informational_only`.
- `lizAssetNeeds`: explicit requirements for Liz.
- `agentAssetPlan`: what the image/video agent can generate.
- `rationale`: why this route fits the story.
- `risk`: `low`, `medium`, or `high`
- `effort`: 1-5
- `selected`: boolean

### Asset Request

An explicit production obligation.

Fields:
- `storyId`
- `routeId` optional
- `slideId` optional
- `owner`: `liz` or `agent`
- `kind`: `screenshot`, `receipt`, `product`, `reference`, `voice`, `face`,
  `generated_image`, `generated_video`, `other`
- `label`
- `instructions`
- `required`: boolean
- `status`: `needed`, `supplied`, `generating`, `selected`, `waived`
- `filePath` optional
- `createdAt`, `updatedAt`

### Post Draft

The Buffer-like publishable object.

Fields:
- `storyId`
- `platform`, `format`
- `caption`
- `hashtags`
- `coverText`
- `postingNotes`
- `status`: `draft`, `awaiting_approval`, `ready`, or `posted`
- `scheduleIntent` optional: `next_available`, `prioritize`, `date_time`,
  `manual_now`
- `updatedAt`

## User Experience

### Default Screen: All Posts

The first screen is a Buffer-like all-content dashboard.

Tabs:
- `Ideas`: uncommissioned story cards and creative brief route choices.
- `Drafts`: commissioned stories moving through angle, writing, legal, design,
  recording, or production.
- `Approvals`: Gate 1 and Gate 2 decisions.
- `Ready`: packaged posts waiting for manual publishing.
- `Posted`: posted/rated stories.

Each card shows:
- thumbnail or preview placeholder
- title
- platform and format
- selected route or current post type
- next action
- asset state: "2 owed by Liz", "agent generating", "ready"
- short caption/script preview when available

Filters:
- channel/platform
- story status
- business job
- "needs me"

### Story Studio

Clicking a post opens the working studio.

Top:
- title, job, platform, format, current status, brain version
- next action ribbon

Left column:
- research summary
- route options
- selected route state
- claims ledger summary

Center:
- post composer and preview
- script sections / carousel slides / final cut
- generated visual candidates in design stage

Right column:
- asset requests
- generation manifest and costs
- approval controls
- manual posting package when ready

### Creative Brief Flow

The story desk creates 2-4 format routes for every idea. Liz selects one route.
Route selection patches the story's `angle`, `platform`, and `format`, and
creates any required Liz-owned asset requests.

From there, Liz can:
- send the selected route to the angle room
- send it straight to draft
- spike the story

### Asset Inbox

A dedicated "Needs Me" screen batches all Liz-owned production obligations:
- screenshots/receipt crops
- product/reference images
- voice/intro recordings
- final approvals

Every request has instructions, status, source story, and upload/attach
controls where applicable.

### Ready Queue

Ready posts show:
- final video or image set
- caption
- hashtags
- cover text
- posting notes
- files in `media-vault`
- manual posting checklist
- `Mark posted`

## Agent Changes

### Story Desk

In addition to filing story cards, the story desk must output a creative brief
and route options. It should lean into post craft: format choice, asset needs,
visual treatment, and whether the agent can generate the required art.

### Writers' Room

Writers receive the selected format route. They draft to the route rather than
only to a loose story angle.

### Production and Publishing

Production continues to use selected design assets. Publishing writes the
post package and now also saves a `postDraft` row for dashboard display.

### Post Craft Review Loop

> **2026-07-16 update:** the review loop below is now OPTIONAL and never blocks
> Gate 2. Approval happens by inspecting the rendered deck itself in the
> deck-first Final stage (numbered slides, per-slide fix back into visuals).
> Only outstanding required asset requests still block approval. The form and
> its validation rules remain for when a documented pass is wanted.

Every Tier 2, Tier 3, and Tier 4 post should carry a short review log before
Gate 2 approval. The review loop is defined in the Brain doc
`post-craft-review-loop` and checks:

- format lock
- first-frame hook
- message spine
- app as resolver
- asset truth
- voice and compliance
- pixel and motion pass
- platform-native pass

Any amber or red gate must produce a concrete revision instruction. The goal
is not more discussion; it is a loop that keeps improving the artifact until
the viewer can see the pain point, the evidence check, and the app's practical
job without effort.

Review passes with amber or red gates cannot be saved with empty or generic
revision notes. The revision instruction must name the concrete fix that moves
the next gate toward green, such as the exact missing asset, scene range, hook
rewrite, app-demo replacement, or pixel issue to rerender.

Any non-ready review pass (`pending`, `revise`, or `blocked`) must also name
one concrete next action: either a specific revision instruction or one specific
next asset. The loop should not allow status-only review logs.

If multiple gates are amber or red, each open gate must have its own concrete
fix. The shared revision summary is not enough; the app-resolver, hook, message,
asset truth, voice, pixel, and platform issues must remain separable.

Post Studio must show the selected route's review pass history, not only the
latest status. Each pass should expose its blockers, revision instruction, next
asset, and artifact or stills evidence so Liz can see whether the post is
actually improving through the loop.

When Liz changes the selected route, the review form must reset to that route's
next pass number and show only that route's sorted pass history. Draft review
state from another route must not carry across.

All Posts review badges must also be scoped to the selected route. A review from
an unselected route must not make the active route look ready, blocked, or
revised. If the selected route has no review pass, the card should show no
review badge rather than falling back to a different route's pass.

Review passes must be saved against the selected route. The app should reject
unscoped review passes when a story has format routes, and it should reject any
route ID that does not belong to the story. Otherwise a stale or unrelated pass
can falsely unlock Gate 2.

Gate 2 approval is blocked for Tier 2+ selected routes until the latest review
for that route is marked `ready`, every gate is green, and the review has a
rendered artifact path, contact sheet or reviewed stills path, plus a complete
proof pack:

- rendered artifact path
- contact sheet or reviewed stills path
- first-frame proof
- hook promise and payoff
- one-sentence message spine
- app-resolver evidence
- single CTA evidence
- pixel/motion evidence from rendered stills or artifact review
- final viewer action

Gate 2 is also blocked when the selected route, or the story-level production
state, still has required asset requests with status `needed` or `generating`.
Those assets must be supplied, selected, or waived before final approval. Story-
level required assets still block legacy Gate 2 stories that do not have a
selected route.

A ready review pass cannot be saved while non-review required assets are still
`needed` or `generating`. Review-generated `Review asset:` requests are the
exception: a later ready pass waives open `needed` or `generating` review
requests for the route, including legacy story-level review requests.

Readiness checks must enumerate every named gate. A partial or legacy gate
object is not ready just because all of its present values are green. A review
that still names a next asset is also not ready, even if its stored decision is
`ready`.

Tier 1 fast-lane checks may still use the loop without being blocked by it.

If a review pass names `Next asset needed`, Post Studio creates a single
Liz-owned Asset Inbox request for the selected route. A later `ready` review
waives open review-generated requests for that route, keeping the inbox focused
on the current blocker.

When Gate 2 is blocked, the approval panel must list the exact missing review
items: absent review pass, non-ready decision, open gates, missing rendered
artifact path, missing contact sheet or reviewed stills path, missing proof
fields, or the next asset needed. Liz should not have to infer the blocker from
the review log.

## Implementation Plan

1. Add schema tables for creative briefs, format routes, asset requests, and
   post drafts.
2. Add Convex mutations/queries in `design.ts` for route selection, production
   dashboard data, asset request management, and post draft storage.
3. Extend `story-desk.md` output to include creative brief route options.
4. Update `runner.mjs` to save briefs/routes after story creation and to pass
   the selected route to the writers' room.
5. Update packaging to save a `postDraft`.
6. Replace the default dashboard with a Buffer-like All Posts and Story Studio
   UI while preserving the legacy Floor board as a Pipeline view.
7. Add an Asset Inbox view that combines `assetRequests` with existing
   recording queue behavior.
8. Verify with `npm run typecheck`, `npm run build`, and browser inspection of
   the Vite app.

## Review Elements

Use these checks before considering the implementation complete:

- Every idea card can show at least one route, even if it is a fallback route
  generated from older data.
- Selecting a route visibly updates platform/format/angle.
- Liz-owned asset requests appear in the Asset Inbox and on the story.
- Gate 1 and Gate 2 controls still work.
- Design Studio still supports prompt editing, image generation, attaching
  assets, selecting winners, and sending to assembly.
- Ready posts show the actual caption, hashtags, cover text, files, and final
  media before `Mark posted`.
- Existing stories without creative briefs still render.
- Tier 2+ posts have a review-loop record, and amber/red gates have clear
  revision instructions before Gate 2 approval.
- Gate 2 approval is blocked for Tier 2+ selected routes unless the latest
  proof-backed review is ready and green.
- TypeScript and production build pass.
